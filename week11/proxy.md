# 프록시 패턴 (Proxy Pattern)

> 구조 패턴 (Structural Pattern)

## 의도

**프록시**는 다른 객체에 대한 **대체 또는 자리표시자**를 제공하는 구조 디자인 패턴이다.  
프록시는 원래 객체에 대한 접근을 제어하므로, 요청이 원래 객체에 전달되기 전 또는 후에 무언가를 수행할 수 있다.

즉, **진짜 객체를 대신하는 역할**. 클라이언트는 프록시를 통해 실제 객체에 접근한다.

---

## 문제

방대한 시스템 자원을 소비하는 거대한 객체가 있는데, 이 객체는 항상 필요한 것은 아니다.  
(예: 매번 DB 쿼리를 날리는 객체)

- **지연 초기화**를 구현하려면 해당 코드가 모든 클라이언트에 중복된다.
- 클래스가 타사 라이브러리의 일부라 직접 수정이 불가능한 경우도 있다.

## 해결책

원래 서비스 객체와 **같은 인터페이스**를 구현하는 새 프록시 클래스를 생성한다.  
클라이언트로부터 요청을 받으면 프록시가 필요한 처리(캐싱, 접근 제어 등)를 수행한 뒤 실제 서비스 객체에 위임한다.

클라이언트 입장에서는 프록시인지 실제 객체인지 구분할 필요가 없다.

---

## 구조

```
Client ──→ «interface»
            ServiceInterface
            + operation()
                 △
        ┌────────┴────────┐
      Proxy             Service
  - realService: Service    ...
  + checkAccess()       + operation()
  + operation()
        │
  if (checkAccess()) {
      realService.operation()
  }
```

| 구성 요소 | 역할 |
|-----------|------|
| **ServiceInterface** | 서비스의 인터페이스 선언. 프록시가 서비스로 위장할 수 있도록 이 인터페이스를 따라야 함 |
| **Service** | 실제 비즈니스 로직을 제공하는 클래스 |
| **Proxy** | 서비스 객체에 대한 참조 필드를 가짐. 요청 처리(초기화 지연, 로깅, 캐싱 등) 후 서비스에 위임. 서비스의 수명 주기를 관리하기도 함 |
| **Client** | 같은 인터페이스를 통해 서비스와 프록시를 동일하게 다룸 |

---

## 프록시의 종류 (접근 제어 방식)

### 1. 원격 프록시 (Remote Proxy)
- 서비스 객체가 **원격 서버**에 있는 경우
- 프록시의 메소드를 호출하면 그 호출이 네트워크로 전달되어 원격 객체의 메소드가 호출된다
- 네트워크와의 복잡한 세부 사항을 프록시가 처리함

### 2. 가상 프록시 (Virtual Proxy)
- **생성 비용이 큰 객체**를 대신함
- 진짜 객체가 실제로 필요한 순간까지 객체 생성을 미룸 (지연 초기화)
- 객체 생성이 끝나면 실제 객체에 직접 요청을 전달함

### 3. 보호 프록시 (Protection Proxy)
- **접근 권한**이 필요한 자원으로의 접근을 제어
- 클라이언트의 자격 증명이 정해진 기준과 일치하는 경우에만 서비스 객체에 요청을 전달

### 4. 캐싱 프록시 (Caching Proxy)
- 클라이언트 요청 결과를 캐시하고 수명 주기를 관리
- 동일한 결과를 반복 생성하는 요청에 캐싱을 적용. 요청 매개변수를 캐시 키로 사용

### 5. 로깅 프록시 (Logging Proxy)
- 서비스 객체에 대한 요청들의 기록을 유지
- 각 요청을 서비스에 전달하기 전에 로깅

### 6. 스마트 참조 (Smart Reference)
- 사용 클라이언트가 없을 때 거대한 객체를 해제해야 하는 경우
- 프록시가 서비스 객체 참조를 얻은 클라이언트들을 추적하고, 리스트가 비면 객체를 닫고 시스템 자원을 확보

---

## 장단점

### 장점
- 클라이언트가 알지 못하는 상태에서 서비스 객체를 제어할 수 있다
- 클라이언트가 신경 쓰지 않을 때 서비스 객체의 수명 주기를 관리할 수 있다
- 서비스 객체가 준비되지 않았거나 사용할 수 없는 경우에도 작동한다
- **개방/폐쇄 원칙**: 서비스나 클라이언트를 변경하지 않고도 새 프록시를 도입할 수 있다

### 단점
- 새로운 클래스를 많이 도입해야 해서 코드가 복잡해질 수 있다
- 서비스의 응답이 늦어질 수 있다

---

## 실제 상황 비유

> 신용 카드 → 은행 계좌의 프록시  
> 은행 계좌 → 현금의 프록시

둘 다 `Payment` 인터페이스를 구현하며 결제에 사용 가능하다.  
소비자는 현금을 들고 다닐 필요 없고, 상점 주인은 현금 분실 위험이 없다.  
→ 같은 인터페이스, 내부 구현만 다름

---

## 다른 패턴과의 관계

| 패턴 | 인터페이스 | 특징 |
|------|-----------|------|
| **어댑터** | 다른 인터페이스 제공 | 호환되지 않는 인터페이스를 변환 |
| **프록시** | 같은 인터페이스 제공 | 접근 제어 / 수명 주기 관리 |
| **데코레이터** | 향상된 인터페이스 제공 | 기능 추가, 합성이 클라이언트에 의해 제어 |
| **퍼사드** | 단순화된 인터페이스 제공 | 복잡한 서브시스템을 숨김 |

- **프록시 vs 데코레이터**: 구조는 비슷하나 의도가 다름. 프록시는 자체적으로 서비스의 수명 주기를 관리하지만, 데코레이터의 합성은 항상 클라이언트가 제어한다.
- **프록시 vs 퍼사드**: 퍼사드는 복잡한 시스템을 보호하고 자체 초기화한다는 점에서 유사하지만, 프록시는 서비스와 동일한 인터페이스를 가져 상호 교환이 가능하다.

## 실제 사례: 웹 배포에서의 프록시 처리

### CORS란?

**CORS** = Cross-Origin Resource Sharing (교차 출처 리소스 공유)

브라우저가 강제하는 보안 정책으로, **출처(Origin)가 다른 서버에 요청할 때 브라우저가 이를 차단**한다.  
여기서 출처란 `프로토콜 + 도메인 + 포트`의 조합이다.

```
http://localhost:5173  ≠  http://localhost:8080   → 포트가 다르면 다른 출처
```

---

### CORS가 필요한 이유: CSRF 공격

**CSRF** = Cross-Site Request Forgery (사이트 간 요청 위조)

브라우저는 쿠키(인증 정보)를 요청 시 **자동으로 함께 전송**한다.  
CSRF는 이 특성을 악용해, 사용자 몰래 인증된 요청을 다른 사이트에서 날리는 공격이다.

```
1. 사용자가 bank.com에 로그인 → 브라우저에 쿠키(세션) 저장
2. 사용자가 evil.com 방문
3. evil.com의 JS가 실행:
   fetch("https://bank.com/api/transfer?to=hacker&amount=1000000")
4. 브라우저가 bank.com 쿠키를 자동으로 붙여서 전송
5. bank.com은 "인증된 요청"으로 판단 → 송금 실행
```

사용자는 아무것도 하지 않았지만 인증된 요청이 전송된다.  
CORS는 `evil.com → bank.com` 크로스 오리진 요청을 브라우저 단에서 차단함으로써 이 공격을 막는다.

---

### 왜 서버 간 통신에는 CORS가 적용되지 않는가

CORS가 막는 핵심 위협은 **"브라우저가 쿠키를 자동으로 붙여 보내는 것"** 이다.  
서버끼리 통신할 때는 이 메커니즘 자체가 존재하지 않는다.

```
브라우저  → bank.com : 쿠키 자동 전송 O → CSRF 위협 존재 → CORS 필요
서버A    → bank.com : 쿠키 자동 전송 X → CSRF 위협 없음 → CORS 불필요
```

서버는 요청 시 쿠키를 자동으로 붙이지 않으며, 인증 정보를 명시적으로 코드에서 설정해야 한다.  
따라서 공격자가 "사용자인 척" 서버를 통해 요청을 날리는 것이 불가능하다.  
이런 이유로 CORS는 **브라우저만** 강제하는 정책이다.

> CORS는 서버끼리의 통신에는 적용되지 않는다. **브라우저만** 이 정책을 강제한다.

### 환경별 구성 및 흐름 비교

#### 개발 환경 (Vite Dev Server Proxy)

프론트(`localhost:5173`)와 백엔드(`localhost:8080`)를 따로 띄울 때 CORS 문제가 발생한다.  

#### 배포 환경 (Nginx Reverse Proxy)

실제 배포에서는 Nginx 하나로 프론트 정적 파일 서빙 + 백엔드 라우팅을 동시에 처리한다.

```nginx
server {
    listen 80;

    # 프론트엔드 정적 파일
    location / {
        root /var/www/dist;
        try_files $uri $uri/ /index.html;
    }

    # API 요청은 백엔드로 프록시
    location /api/ {
        proxy_pass http://localhost:8080;
    }
}
```

```
브라우저 → Nginx (443 또는 80)
               ├── /         → 정적 파일 서빙 (프론트)
               └── /api/     → localhost:8080 (백엔드 WAS)
```

#### 흐름 비교

```
[개발]  브라우저 → Vite dev server (Proxy) → Spring API
[배포]  브라우저 → Nginx (Proxy)            → Spring API
                                             → 정적 파일
```

개발 환경의 `proxy` 설정이 배포 환경에서는 Nginx 설정으로 대체되는 구조다.

---

### 프록시가 CORS를 우회하는 원리

프록시가 CORS를 "해결"하는 것이 아니라, **브라우저가 직접 다른 출처에 요청하지 않도록 중간에 끼어드는 것**이다.

```
[직접 요청 - CORS 발생]
브라우저(5173) ────────────────→ 백엔드(8080)
                  ← 브라우저가 차단

[프록시 경유 - CORS 없음]
브라우저(5173) → 프록시(5173) → 백엔드(8080)
  같은 출처 ✓      서버→서버 통신 (CORS 미적용) ✓
```

브라우저 입장에서는 자신과 같은 출처(`5173`)에만 요청한 것이므로 CORS 정책에 걸리지 않는다.  
프록시→백엔드 구간은 서버 간 통신이므로 CORS 자체가 존재하지 않는다.

### 총 정리

| 구분 | 내용 |
|------|------|
| 프록시의 본래 목적 | 접근 제어, 캐싱, 로깅, 부하분산 등 |
| 개발 환경에서 쓰는 이유 | CORS 우회가 주목적 |
| 배포 환경에서 쓰는 이유 | 단일 진입점 제공, CORS 우회, SSL 처리, 로드밸런싱 |
| CORS 우회가 되는 이유 | CORS는 브라우저 정책 → 서버→서버 통신엔 적용 안 됨 |
| 패턴 관점 | 클라이언트(브라우저)는 프록시 뒤에 뭐가 있는지 모름 → 프록시 패턴과 동일한 구조 |
