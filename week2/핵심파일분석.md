# 최소 필수독해 코드 분석
## 핵심 데이터 흐름 요약

```
사용자 입력
    │
    ▼
server.ts (POST /session/:id/message)
    │
    ▼
prompt.ts → createUserMessage() → [DB 저장]
    │
    ▼
prompt.ts → runLoop() ──────────────────────────┐
    │                                             │
    ▼                                             │ loop
llm.ts → streamText()                            │
    │                                             │
    ▼                                             │
processor.ts → handle events                     │
    │                                             │
    ├──[tool-call]──► tool/bash.ts / task.ts      │
    │                    └──► permission/index.ts  │
    │                           └──[ask]──► ⏸️ pause│
    │                           └──[reply]──► ▶️ resume
    │                                             │
    ├──[continue]──────────────────────────────►──┘
    │
    └──[stop]──► lastAssistant() → 반환
```

---

## 목차

- session/prompt.ts — ReAct 루프 본체
- session/processor.ts — 스트리밍 이벤트 처리
- session/llm.ts — LLM 호출 레이어
- tool/tool.ts — Tool 인터페이스 정의
- tool/bash.ts — Bash 실행 도구
- tool/task.ts — 멀티에이전트 오케스트레이션
- provider/provider.ts — LLM 프로바이더 관리
- permission/index.ts — Human-in-the-Loop
- server/server.ts — IPC 서버
-  session/message-v2.ts — 메시지/파트 타입 시스템

### session/prompt.ts
- ReAct 루프 관련이므로 루프와 관련된 함수 runLoop가 핵심
#### runLoop의 흐름 요약
- ubtask/compaction/overflow를 먼저 소진하고 나서야 실제 LLM을 호출하는 우선순위 기반 이벤트 드리븐 루프라는 점이 핵심
```
while(true)
  │
  ├─[1] msgs 로드 + 상태 파악
  ├─[2] 이미 완료됐으면? → break
  │
  ├─[3] subtask 대기중? → handleSubtask() → continue (LLM 스킵)
  ├─[3] compaction 대기중? → compaction.process() → continue
  ├─[3] 토큰 오버플로? → compaction.create() → continue
  │
  ├─[4] 빈 msg 생성 → DB 저장 → processor 생성
  ├─[5] 시스템 프롬프트 + 도구 조립
  ├─[6] handle.process() → LLM 스트리밍 + 도구 실행
  │       └─ tool-call 발생 시 permission.ask()로 ⏸️ 가능
  │
  ├─[7] result="stop"    → break
  ├─[7] result="compact" → 압축 예약 → continue
  └─[7] result="continue"→ 다음 iteration

```

### session/processor.ts
- 핵심 함수는 LLM 스트리밍 루프 전체 골격 함수인 process(), 그리고 각 이벤트 타입별 실제 처리 로직인 handleEvent() 이렇게다.
#### 흐름 요약
```
runLoop()
  └─► handle.process()          ← 443번: LLM 호출 + 이벤트 루프
          └─► handleEvent(event) ← 107번: 이벤트 하나씩 처리
                  ├─ text-delta  → msg에 텍스트 추가
                  ├─ tool-call   → 도구 실행
                  └─ finish      → "stop"/"continue" 결정

```
__process()__
- process() = llm.stream() 실행 + handleEvent()로 이벤트 처리 + 재시도 + "stop"/"continue"/"compact" 반환

### session/llm.ts
핵심 함수는 `stream()`. Vercel AI SDK `streamText()`를 호출하기까지의 준비 과정이 전부.

#### 흐름 요약
```
stream()
  ├─ 프로바이더/설정/인증 병렬 로드
  ├─ 시스템 프롬프트 조립 (agent.prompt + env + instructions)
  ├─ 옵션 병합 (base ← model ← agent ← variant 순)
  ├─ Plugin 훅 실행 (chat.params, chat.headers)
  ├─ resolveTools() → permission으로 비활성 도구 필터링
  ├─ LiteLLM 버그 우회 → _noop 도구 주입
  └─ streamText({ model, tools, messages, ... })
       └─ experimental_repairToolCall: "Bash" → "bash" 자동 수정
```

### tool/tool.ts
`Tool.define()` 팩토리가 핵심. 모든 도구의 `execute()`를 자동으로 래핑.

#### 자동 래핑 내용
- **Zod 유효성 검사**: 실패 시 LLM이 읽을 수 있는 오류 메시지로 변환
- **출력 truncation**: 너무 긴 출력을 파일로 저장하고 경로 반환

---

### tool/bash.ts
핵심 함수 3개: `collect()` → `ask()` → `run()`.

#### 흐름 요약
```
execute()
  ├─ parse() → tree-sitter로 AST 파싱
  ├─ collect() → AST 스캔해서 접근할 파일 경로 정적 추출
  ├─ ask() → permission 요청 (외부 경로 / 명령어 패턴)
  └─ run() → spawn() + 실시간 출력 스트리밍 + 2분 타임아웃
```
- `collect()`가 핵심: 명령어를 실행하기 전에 어떤 파일에 손댈지 미리 파악하는 정적 분석

### tool/task.ts
핵심 함수는 `execute()`. 재귀적으로 `SessionPrompt.prompt()`를 호출하는 것이 전부.

#### 흐름 요약
```
execute()
  ├─ permission.ask("task") → 승인 대기
  ├─ Session.create({ parentID }) → 자식 세션 생성
  └─ SessionPrompt.prompt({ sessionID: 자식 }) → 재귀 호출
       └─ 자식 세션에서 runLoop()가 그대로 실행됨
```
- `task_id` 파라미터로 이전 서브에이전트 세션을 재개할 수 있음

### provider/provider.ts
핵심 함수 2개: `get()` (모델 인스턴스 반환) + `init()` (초기화 파이프라인).

#### init() 순서
```
BUNDLED_PROVIDERS → ModelsDev API → 환경변수 → config → auth
→ Plugin 훅 → CUSTOM_LOADERS (Anthropic/Bedrock/GitLab 특수처리)
→ whitelist → blacklist
```
- `get()`은 단순 맵 조회지만, `init()`의 9단계 파이프라인을 거쳐야 맵이 채워짐

---

### permission/index.ts
핵심 함수 2개: `ask()` (루프 일시정지) + `reply()` (루프 재개).

#### 흐름 요약
```
ask()
  ├─ evaluate() → "deny" → 즉시 에러
  ├─ evaluate() → "allow" → 즉시 통과
  └─ evaluate() → "ask" → Deferred.await() → ⏸️ 블로킹

reply()
  ├─ "once"   → Deferred.succeed() → ▶️ 재개
  ├─ "always" → 규칙 저장 → 대기 중인 동일 패턴 일괄 재개
  └─ "reject" → Deferred.fail() → 같은 세션 전체 거부
```

### server/server.ts
핵심 함수 2개: `ControlPlaneRoutes()` (앱 정의) + `listen()` (서버 실행).

#### 구조 요약
```
ControlPlaneRoutes()
  ├─ 미들웨어: BasicAuth → 로깅 → CORS → gzip
  └─ 라우트: /auth, /doc, /log + WorkspaceRouterMiddleware
       └─ 주요 라우트:
           POST /session/:id/prompt  → ReAct 루프 시작
           GET  /event               → SSE 스트림 (모든 이벤트 실시간 전송)
           POST /permission/:id/reply → Human-in-the-Loop 응답

listen() → Bun.serve() 실행, port=0이면 4096 먼저 시도
```

### session/message-v2.ts
타입 정의 파일. 직접 실행 로직은 없고 `toModelMessages()`가 유일한 핵심 함수.

#### 알아야 할 타입들
- `WithParts` — DB에서 꺼낸 메시지 (info + parts[])
- `ToolPart.state` — `pending → running → completed | error` 상태 머신
- Part 종류: `text`, `reasoning`, `tool`, `step-start/finish`, `patch`, `compaction`, `subtask`

#### toModelMessages()
DB의 `WithParts[]`를 Vercel AI SDK가 받는 `ModelMessage[]`로 변환. `compaction` 파트는 이전 메시지 전체를 요약 텍스트 하나로 대체.
