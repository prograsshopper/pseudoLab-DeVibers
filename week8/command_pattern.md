# 커맨드 패턴 (Command Pattern)

## 1. 개념

**요청을 객체로 캡슐화**하여, 요청을 보내는 쪽(Invoker)과 요청을 처리하는 쪽(Receiver)을 분리하는 **행동(Behavioral) 디자인 패턴**.

> "요청 내역을 객체로 캡슐화해서, 객체를 서로 다른 요청 내역에 따라 매개변수화할 수 있다.
> 이러면 요청을 큐에 저장하거나, 로그로 기록하거나, 작업 취소 기능을 사용할 수 있다."

---

## 2. 구성 요소

```
Client ──creates──▶ ConcreteCommand ──implements──▶ <<Command>>
                         │                              │
                    holds Receiver                  execute()
                         │                           undo()
                         ▼
                      Receiver
                    (action())

Client ──setCommand()──▶ Invoker ──execute()──▶ Command
```

### Client (클라이언트)
- `ConcreteCommand` 객체를 생성하고, `Receiver`를 설정한다.
- 생성한 커맨드 객체를 `Invoker`에 전달한다.

### Invoker (인보커 / 발송자)
- 커맨드 객체를 저장(`setCommand()`)하고, 나중에 `execute()`를 호출해 명령을 실행한다.
- 어떤 `Receiver`가 처리하는지, 어떻게 처리하는지 알 필요가 없다.
- 커맨드 인터페이스를 통해서만 커맨드와 통신한다.

### Command Interface (커맨드 인터페이스)
- 모든 커맨드 객체가 구현해야 하는 인터페이스.
- `execute()`: 행동을 캡슐화하여 `Receiver`의 특정 행동을 처리하도록 지시.
- `undo()`: 실행 취소를 위한 메서드.

### ConcreteCommand (구상 커맨드)
- 특정 행동과 `Receiver`를 연결한다.
- `execute()` 호출 시 `Receiver`의 메서드를 호출해 실제 작업을 처리한다.
- 요청 인수들과 수신자 객체에 대한 참조를 필드로 저장하며, 이 값들은 생성자로 초기화된다.

### Receiver (리시버 / 수신자)
- 실제로 요청을 수행하는 객체. 작업을 어떻게 처리해야 하는지 알고 있다.

---

## 3. 실행 흐름 (인보커 로딩)

```
1. Client  →  ConcreteCommand 생성 (Receiver 포함)
2. Client  →  Invoker.setCommand(command) 호출
3. (나중에) Client  →  Invoker에게 명령 실행 요청
4. Invoker →  command.execute() 호출
5. Command →  receiver.action() 호출
```

---

## 4. NoCommand 객체 (Null Object)

- 인보커에 커맨드가 아직 설정되지 않은 경우를 처리하기 위한 **널 객체(Null Object)** 패턴의 적용.
- 리턴할 객체가 없거나, 클라이언트 쪽에서 `null` 체크를 하지 않아도 되게 하고 싶을 때 활용.
- 아무 동작도 하지 않는 `execute()`를 구현한 커맨드 객체를 기본값으로 설정해두는 방식.

---

## 5. 적용 시나리오

| 상황 | 설명 |
|------|------|
| 작업의 매개변수화 | 메서드 호출을 독립 객체로 전환해 런타임에 커맨드 교체 가능 |
| 실행 예약 / 대기열 | 커맨드를 직렬화하여 파일/DB 저장 후 나중에 복원·실행 가능 |
| 원격 실행 | 네트워크를 통해 커맨드를 전송해 원격으로 작업 실행 가능 |
| 실행 취소(Undo) | 커맨드 기록 스택을 유지해 이전 상태로 되돌리기 가능 |

---

## 6. 구현 방법

1. **커맨드 인터페이스 선언** — 단일 `execute()` 메서드를 포함.
2. **구상 커맨드 클래스 작성** — 수신자 참조와 요청 인수를 필드로 저장, 생성자에서 초기화.
3. **발송자(Invoker) 클래스 식별** — 커맨드를 저장할 필드 추가, 인터페이스를 통해서만 통신.
4. **발송자가 커맨드를 실행하도록 변경** — 수신자에게 직접 요청 대신 `command.execute()` 호출.
5. **클라이언트에서 조립**:
   - Receiver 생성
   - Command 생성 후 Receiver와 연결
   - Invoker 생성 후 Command와 연결

---

## 7. 장단점

### 장점
- **단일 책임 원칙(SRP)** — 작업을 호출하는 클래스와 수행하는 클래스를 분리.
- **개방/폐쇄 원칙(OCP)** — 기존 코드 수정 없이 새로운 커맨드 추가 가능.
- **Undo/Redo 구현 용이** — 커맨드 기록 스택으로 실행 취소/재실행 지원.
- **지연 실행** — 커맨드를 즉시 실행하지 않고 나중에 실행 가능.
- **매크로 커맨드** — 여러 커맨드를 조합해 복잡한 커맨드 구성 가능.

### 단점
- 발송자와 수신자 사이에 **새로운 레이어가 추가**되어 코드가 복잡해질 수 있음.

---

## 8. 관련 패턴 비교

| 패턴 | 발신자-수신자 연결 방식 |
|------|------------------------|
| **책임 연쇄** | 수신자 체인을 따라 순차적으로 전달, 처리자 찾으면 중단 |
| **커맨드** | 발신자와 수신자 간 **단방향 연결** 설립 |
| **중재자** | 직접 연결 제거, 중재자 객체를 통해 **간접 통신** |
| **옵서버** | 수신자가 요청 수신을 **동적으로 구독/취소** 가능 |

### 함께 쓰이는 패턴
- **메멘토(Memento)** + 커맨드: Undo 구현 시 커맨드 실행 전 상태를 메멘토로 저장.
- **프로토타입(Prototype)**: 커맨드 복사본을 기록(history)에 저장할 때 활용.
- **전략(Strategy)** vs 커맨드:
  - 커맨드 — 작업 자체를 객체화, **실행 지연·대기열·원격 전송** 목적.
  - 전략 — 동일 작업을 수행하는 **다양한 알고리즘**을 교환하는 목적.
- **비지터(Visitor)**: 커맨드의 강화 버전으로, 다양한 클래스 객체에 대해 작업 실행 가능.
